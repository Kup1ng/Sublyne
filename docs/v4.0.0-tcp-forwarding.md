# v4.0.0 — TCP forwarding (KCP / QUIC) + modal tunnel form

A plain-language guide to what changed in v4.0.0. ~10 minutes.

## The one-sentence version

Every tunnel can now forward **TCP** (not just UDP), so you can hand your
users **VLESS-TCP / VLESS-WS** configs instead of WireGuard/Hysteria — and
the tunnel create/edit screen is now a **pop-up form** instead of a separate
page.

## Why this exists

Until now a Sublyne tunnel only carried **UDP**: a packet in at the listen
port went out as a UDP packet. That forced you to give users UDP-based
configs (WireGuard, Hysteria). Many users find **VLESS-TCP / VLESS-WS**
easier to set up and share — but those need a *reliable TCP* connection, and
Sublyne's download path is **best-effort**: a spoofed packet that's lost or
badly reordered is gone forever. Raw TCP can't survive that (one lost byte
corrupts the whole stream forever); UDP tolerates it because the inner
protocol (WireGuard) repairs itself.

v4.0.0 solves this by putting a **reliability engine** between the user's TCP
connection and the spoof channel. The engine turns the lossy packet channel
into a solid byte-stream. Two engines are available — **KCP** and **QUIC**.

## The new per-tunnel setting

Each tunnel now has a **Forward protocol**:

- **UDP** *(default)* — exactly today's behavior. Nothing changes.
- **TCP** — the tunnel terminates the user's TCP connection, carries it
  reliably across the spoof channel, and re-opens TCP to your real service on
  the foreign box. When you pick TCP you also choose:
  - **Reliability engine:** **KCP** *(recommended)* or **QUIC**.
  - **Tuning preset:** *Low-latency interactive*, *Balanced / bulk*, or
    *Lossy / aggressive recovery* — sensible defaults per engine.
  - **Advanced** *(optional)* — fine-tune individual numbers; clearing a
    field reverts it to the preset.

These four choices are **shared by both servers** — set the same on the
Iran-side (Client) and foreign-side (Remote) tunnel, exactly like you already
do for the PSK.

### Worked example — KCP

> You run a VLESS-TCP server on the foreign box at `192.0.2.10:443`.
> Create a tunnel with **Forward protocol = TCP**, **Engine = KCP**, preset
> **Balanced / bulk**, forward target `192.0.2.10`, port `443`. Set the
> matching tunnel on the Iran box. Now hand users a normal VLESS-TCP config
> pointing at the Iran box's listen port. Their TCP connection rides KCP over
> the spoof channel; if packets drop, KCP retransmits — the user just sees a
> working connection.

### Worked example — QUIC

> Same as above but **Engine = QUIC**. QUIC multiplexes many user
> connections over one encrypted QUIC link and is widely supported. It needs
> the tunnel **MTU ≥ 1252** (QUIC packets are ≥1200 bytes); the panel enforces
> this. On very hostile ICMP paths KCP is the safer pick (QUIC pads its
> handshake to 1200 bytes).

## The modal tunnel form

Creating or editing a tunnel now opens a **centered pop-up** over the Tunnels
page instead of navigating to a separate page. Same fields, same validation,
same Save behavior. The new Forward-protocol controls appear inside it; the
form scrolls internally so the Save/Cancel buttons always stay visible. The
old `/tunnels/new` and `/tunnels/<id>` pages are gone — the dashboard's
"edit" and "new" buttons open the pop-up instead.

## How your existing tunnels migrated

**Zero action required.** On first start of v4.0.0, a database migration adds
the new columns and sets every existing tunnel to **forward_protocol = udp** —
which is byte-for-byte the behavior you have today. Nothing about your running
UDP tunnels changes.

## Wire-format change + upgrade procedure

- The seal/spoof envelope is **unchanged**, so a **UDP-forwarding tunnel on
  v4.0.0 still interoperates with a v3.x peer**. You can upgrade one box at a
  time for UDP tunnels.
- A **TCP-forwarding tunnel requires BOTH ends on v4.0.0+** (the reliability
  engine doesn't exist on v3). To switch a tunnel to TCP: upgrade both its
  boxes to v4.0.0, then set forward_protocol = tcp + an engine on each.

That's why this is a **major** version: it's a significant new capability and
TCP tunnels need a coordinated both-ends upgrade.

## Honest caveats

- **Not yet validated on real hardware.** Each engine is proven by an
  automated test that pushes data through a simulated channel that drops 10%
  and reorders 5% of packets and confirms the bytes survive intact — this runs
  on every build. But the *full* spoof path (raw sockets) needs root, so it
  can't run in CI. **Please validate a `tcp+kcp` and a `tcp+quic` tunnel on
  your two-VPS pair** before relying on them.
- **KCP vs QUIC.** KCP is simpler, more predictable, best on lossy/ICMP paths,
  and has no MTU floor — it's the recommended default. QUIC adds native
  multiplexing and built-in encryption but is heavier, needs MTU ≥ 1252, and
  its loss-based congestion control can back off harder on very lossy links.
- **Multi-port TCP forwarding** is **supported in v4.0.0**: list several
  ports on a TCP tunnel and each port gets its **own independent reliability
  engine** (its own listener/dial, conversation map, and idle reaper), all
  sharing the one seal/upload pipeline and demuxed by the same 2-byte
  application-port tag the UDP multi-port path uses. One slow or busy port
  never stalls the others. The chosen engine, preset, and Advanced tuning
  apply to every port (one tunnel = one engine kind). A **single-port** TCP
  tunnel stays byte-for-byte identical on the wire (no tag). TCP forwarding
  works on **all six** download×upload combinations (UDP-WG, TCP-SOCKS5,
  ICMP-WG, ICMP-SOCKS5, ICMPv6-WG, ICMPv6-SOCKS5).

## Under the hood (one paragraph)

The engine lives entirely at the forward-payload layer in the Rust data plane
(`data-plane/src/forward/`). It consumes and produces opaque datagrams; the
existing seal / HMAC / 1024-slot anti-replay / spoof pipeline keeps carrying
those datagrams unchanged — it never learns they now hold KCP/QUIC framing.
KCP uses one conversation per user connection; QUIC uses one connection per
tunnel with one stream per user connection, driven over a custom
`quinn::AsyncUdpSocket` so it rides the same channel. All crypto is pinned to
the `ring` provider so the single static Linux binary still needs no C
toolchain.
