# Manual acceptance checklist — PRD §11

Items in this checklist need a human reviewer (and in some cases two
real VPS hosts). Tick each box in the Phase 15 PR description.

## #2 — Real WireGuard config + matching tunnel pair + at least one transport E2E

- [ ] On the Iran-side server (Client role): paste a real WireGuard
      config (e.g. the seller's Starlink endpoint) into the panel and
      save it.
- [ ] Create a Client tunnel referencing that WG config; pick a
      `download_transport` of UDP for this first pass.
- [ ] On the foreign server (Remote role): create a matching Remote
      tunnel with the same PSK, ports, transport, MTU.
- [ ] Enable both tunnels.
- [ ] From a third device: send UDP traffic at the Client's
      `local_listen_addr` and confirm the Remote's
      `forward_target` receives it and replies pass back through
      the spoofed download path.
- [ ] Verify the dashboard shows packet activity on both sides.

## #3 — All four spoof transports verified

Run #2 four times, switching `download_transport` on both ends each
time. Capture the dashboard's per-transport packet counter for each.

- [ ] **UDP** envelope — works.
- [ ] **TCP-SYN** envelope — works; verify on Client side that
      `iptables` / `nftables` blocks the kernel RST so the spoofed
      SYN payload is delivered to the dataplane raw socket.
- [ ] **ICMP** envelope — works *if* the path doesn't drop ICMP.
      Many Iranian transit ISPs filter ICMP; if loss is 100 %, mark
      "path-filtered" rather than "broken" and pick a different
      transport.
- [ ] **ICMPv6** envelope — works between two IPv6-reachable hosts.

## #4 — Bandwidth meter ±5 %

- [ ] Start `iperf3 -s -p 5201` on the Remote-side target.
- [ ] Start `iperf3 -c <client-public-ip> -u -p <local_listen_addr>
      -b 50M -t 30` from a third device.
- [ ] Open the dashboard and read the per-tunnel "Bandwidth (up/down)"
      gauges over the same 30 s window.
- [ ] Confirm the panel reading is within ±5 % of iperf3's number.

## #5 — 1000+ sessions + 500 Mbps + panel responsive + CPU < 80 %  *(blocked on perf pass)*

The current single-stream throughput is ~50 Mbit/s. The deferred
performance rework targets ~200 Mbit/s per tunnel × multiple tunnels
= the 1 Gbps aggregate the PRD set. Until that lands, this criterion
is documented as **blocked on perf pass**:

- [ ] Re-run after the perf pass lands. Until then mark BLOCKED.
